Intention (what we are trying to do)

  • Align teams on a single end-to-end workflow (value stream), not a scattered list of AI tools.
  • Make tradeoffs explicit: speed vs review burden vs incident risk vs doc drift.
  • Turn “AI adoption” into measurable delivery learning by pairing the map with DORA definitions and a tiny KPI cadence.
  • Keep it lightweight: labels + weekly export + sprint rollup—no platform project required to start.

Evals + DORA — progress and impact

The value stream map shows where AI fits; this strip shows how we learn whether changes help or hurt delivery. Definitions stay in dora-definitions.md; numbers flow weekly → sprint → experiments with explicit stop rules.

Attribution. DORA is service-level outcomes. AI impact is read by cohorts and trends (for example PR labels such as ai-assisted vs control, or before/after a bounded experiment)—not by relabeling deploy frequency as an “AI score.”

1 · Lock Pilot scope, deploy unit, CFR severity threshold, PR label policy (dora-definitions.md).
→
2 · Weekly Per-service export: DF, LT, CFR, MTTR + cheap leading signals into weekly-dora-export-template.csv.
→
3 · Sprint rollup Medians / rates merged into kpi-tracker.csv; adoption map update in the same retro cadence.
→
4 · Eval + decide experiment-log.csv: hypothesis, guardrails, stop if CFR or flake rate breaches.
→
5 · Narrative Quarterly story: what moved, confounders, what we standardize vs stop (plan phases P0–P3).
DF
Deployment frequency

Successful production deploys per window for the in-scope service or domain.

LT
Lead time for changes

Median time from first commit on the integration path to production deploy.

CFR
Change failure rate

Failed prod deploys ÷ prod deploys, using agreed severity and rollback rules.

MTTR
Mean time to restore

Median detect → restored for incidents attributed to an in-scope production deploy.

P0

Instrumentation — labels, pilot repos, baseline window, dashboards trustworthy.

P1

Observe — weekly cohort trends; document confounders; no causal claim.

P2

Experiment — sprint-sized change, hypothesis on 1–2 metrics, hard stop rules.

P3

Scale — bake in standards; automate weekly digest; reduce measurement tax.
